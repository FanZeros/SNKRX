# SNKRX Engine Adapter: Comprehensive Missing/Broken API Report

Generated by analysis of all 9 game code files vs all 48 engine adapter files.

**Note**: Original SNKRX engine files at `/tmp/snkrx_extract/` were NOT found on disk,
so the comparison is based on what the game code expects vs what the adapter implements.

---

## SEVERITY LEGEND

- **CRITICAL** -- Will cause immediate runtime crash (`attempt to call a nil value` / `attempt to index a nil value`)
- **HIGH** -- Will cause incorrect behavior or silent failure that breaks gameplay
- **MEDIUM** -- Functionality degraded but game may still partially work
- **LOW** -- Cosmetic or minor behavioral difference

---

## 1. STEERING / AI MOVEMENT METHODS (CRITICAL)

The game code heavily uses steering behaviors via the Physics mixin. The `Steering` class
exists as a standalone object in `scripts/engine/game/steering.lua`, but the Physics mixin
in `scripts/engine/game/physics.lua` does NOT implement the convenience methods that game
code calls directly on game objects.

### 1a. `set_as_steerable(v, max_force, max_turn, separation)`

**Severity**: CRITICAL
**Status**: NOT IMPLEMENTED in Physics mixin
**Called in**:
- `player.lua:3479` -- `self:set_as_steerable(self.v, 2000, 4*math.pi, 4)` (Projectile init)
- `player.lua:3553` -- `self:set_as_steerable(self.v, 2000, 4*math.pi, 4)` (Projectile init)
- `player.lua:3886` -- `self:set_as_steerable(self.v, 400, math.pi, 1)` (Critter init)
- `enemies.lua:15` -- `self:set_as_steerable(self.v, 1000, 2*math.pi, 2)` (Seeker init)
- `enemies.lua:171` -- `self:set_as_steerable(self.v, 2000, 4*math.pi, 4)` (EnemyProjectile init)
- `enemies.lua:859` -- `self:set_as_steerable(self.v, 400, math.pi, 1)` (EnemyCritter init)

**What it should do**: Initialize steering state on the physics body: set max_speed, max_force,
max_turn_rate, separation_radius. Creates internal steering accumulator. Sets initial velocity
from the `v` parameter (speed along current angle).

### 1b. `seek_point(x, y)`

**Called in**:

## 2. `graphics.set_line_width(width)` (HIGH)

**Severity**: HIGH
**Status**: NOT IMPLEMENTED as module-level function
**Called in**:
- `buy_screen.lua:639` -- `graphics.set_line_width(5)`
- `buy_screen.lua:642` -- `graphics.set_line_width(1)`

**Details**: The `Graphics:init_module()` function in `graphics.lua` creates module-level
functions for `rectangle`, `circle`, `line`, etc., but `set_line_width` is not among them.
Individual drawing methods internally call `nvgStrokeWidth()`, but the game code expects
to be able to set a persistent line width that affects subsequent draw calls.

**Fix**: Add to `Graphics:init_module()`:
```lua
self.set_line_width = function(w)
  if vg then nvgStrokeWidth(vg, w) end
end
```

## 4. `draw_with_mask()` -- DEGRADED IMPLEMENTATION (MEDIUM)

**Severity**: MEDIUM
**Status**: IMPLEMENTED but non-functional (stub that skips masking)
**Called in**:
- `shared.lua` -- `shared_draw()` uses it for star pattern masking
- `arena.lua` -- uses it for visual effects

**Current behavior**: Just calls `draw_fn()` without applying the mask.
The mask_fn is completely ignored. This means visual effects that rely on
stencil masking (star patterns, glow masks) will render incorrectly --
the masked content will be visible everywhere instead of only within the mask shape.

**Impact**: Purely visual -- stars will draw without their diamond/shape masking.
Game is playable but visually incorrect.

## 5. CANVAS -- STUB IMPLEMENTATION (MEDIUM)

**Severity**: MEDIUM
**Status**: IMPLEMENTED as no-op stubs
**Defined in**: `scripts/engine/graphics/canvas.lua`

**Game code usage** (from `shared.lua`):
- `Canvas(gw, gh)` -- creates game canvas, shadow canvas, etc.
- `canvas:draw_to(function() ... end)` -- redirects drawing to off-screen buffer
- `canvas:draw(x, y, 0, sx, sy)` -- blits canvas to screen with scaling
- `canvas:set()` / `canvas:unset()` -- push/pop render target
- `canvas:clear()` -- clear canvas contents

**Current behavior**: `draw_to()` calls the action immediately (no off-screen buffer).
`draw()` and `draw2()` are no-ops. The entire multi-pass rendering pipeline (game canvas ->
shadow canvas -> final composite) is effectively collapsed into a single pass.

**Impact**: The design-resolution scaling (gw=480, gh=270 -> screen via sx, sy) is broken.
Shadow/glow post-processing effects are lost. The game renders directly at physical
resolution instead of at the virtual 480x270 then scaling up.

## 8. `Font:get_text_width()` -- POTENTIAL nvgTextBounds ISSUE (MEDIUM)

**Severity**: MEDIUM
**Status**: IMPLEMENTED but may have API mismatch
**Defined in**: `scripts/engine/graphics/font.lua`

**Issue**: The code calls `nvgTextBounds(vg, 0, 0, tostring(text), nil, bounds)` with
a Lua table `bounds = {0, 0, 0, 0}` as the output parameter. NanoVG's C API expects
a `float[4]` pointer. Whether this works depends on the UrhoX Lua binding -- if it
does not write back into the Lua table, all text width measurements will return 0,
breaking the entire Text system (character positioning, alignment, wrapping).

**Impact if broken**: Text rendering would have zero-width characters, causing all
characters to stack at the same position.

**Needs verification**: Test `Font:get_text_width("test")` at runtime to confirm it
returns a non-zero value.

### Fix #1-5: Steering methods in Physics mixin

Add the following to `scripts/engine/game/physics.lua`. The Steering class already
exists in `steering.lua` -- these methods bridge between the Physics mixin's
direct-on-gameobject API and the Steering class.

```lua
-- In Physics mixin:

function Physics:set_as_steerable(initial_speed, max_force, max_turn_rate, separation_radius)
  self.max_speed = initial_speed or 100
  self.max_force = max_force or 1000
  self.max_turn_rate = max_turn_rate or math.pi
  self.separation_radius = separation_radius or 16
  self.steering = Steering(self)
  -- Set initial velocity along current angle
  local vx = math.cos(self.r or 0) * self.max_speed
  local vy = math.sin(self.r or 0) * self.max_speed
  self:set_velocity(vx, vy)
end

function Physics:seek_point(x, y)
  if not self.steering then return end
  local force = Steering.seek(self, x, y, self.max_speed)
  if force then
    local fx, fy = force.x, force.y
    local len = math.sqrt(fx*fx + fy*fy)
    if len > self.max_force then
      fx = fx / len * self.max_force
      fy = fy / len * self.max_force
    end
    self:apply_force(fx, fy)
  end
end

function Physics:steering_separate(radius, group_or_classes)
  if not self.steering then return end
  local neighbors = {}
  if type(group_or_classes) == 'table' then
    if group_or_classes.objects then
      -- It's a group's objects list
      neighbors = group_or_classes
    else
      -- It's a list of classes
      for _, cls in ipairs(group_or_classes) do
        if self.group then
          local objs = self.group:get_objects_by_class(cls)
          for _, o in ipairs(objs) do table.insert(neighbors, o) end
        end
      end
    end
  end
  local force = Steering.separation(self, neighbors, radius, self.max_speed)
  if force then
    self:apply_force(force.x, force.y)
  end
end

function Physics:apply_steering_force(magnitude, angle)
  local fx = math.cos(angle) * magnitude
  local fy = math.sin(angle) * magnitude
  self:apply_force(fx, fy)
end

function Physics:apply_steering_impulse(magnitude, angle, duration)
  -- Apply force over duration using a timer
  local fx = math.cos(angle) * magnitude
  local fy = math.sin(angle) * magnitude
  if self.t then
    self.t:during(duration, function(dt)
      self:apply_force(fx, fy)
    end)
  else
    self:apply_impulse(magnitude, angle)
  end
end
```
